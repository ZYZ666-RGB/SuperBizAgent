关键表关系设计深度解析
设计 ① 用户-角色多对多 → sys_user_role 中间表
一个用户可以拥有多个角色，一个角色可以分配给多个用户
例：用户 teacher01 可同时拥有 TEACHER 和 ADMIN 两个角色
CREATE TABLE sys_user_role (
user_id BIGINT NOT NULL, -- 外键 → sys_user.id
role_id BIGINT NOT NULL, -- 外键 → sys_role.id
UNIQUE (user_id, role_id), -- 防重复绑定
FOREIGN KEY (user_id) REFERENCES sys_user(id),
FOREIGN KEY (role_id) REFERENCES sys_role(id)
);
设计 ② 试卷-题目多对多 → paper_question 中间表
一张试卷包含多道题目，一道题可被多个试卷复用
例：题目"事务ACID特性"可同时出在期中考和期末考中
CREATE TABLE paper_question (
paper_id BIGINT NOT NULL, -- 外键 → paper.id
question_id BIGINT NOT NULL, -- 外键 → question.id
question_score INT NOT NULL, -- 本题在此试卷中的分值
sort_order INT NOT NULL, -- 题目排序
UNIQUE(paper_id, question_id), -- 防重复加入
FOREIGN KEY(paper_id) REFERENCES paper(id) ON DELETE CASCADE
);
设计 ③ 答题快照 — answer_record 的快照字段
question_title_snapshot TEXT — 保存考试时的题干内容
option_snapshot TEXT — 保存考试时的选项（JSON格式）
correct_answer_snapshot VARCHAR — 保存考试时的正确答案
⚠ 原因：教师后续可能修改题目，快照保护历史考试记录独立不变
设计 ④ 关键唯一约束 — 防止重复数据
exam_record.uk(student_id, paper_id)
→ 一名学生对同一张试卷只允许一条考试记录
answer_record.uk(exam_id, question_id)
→ 同一次考试中，同一道题不重复保存答案
CASCADE 说明：删除试卷时自动清理 paper_question / exam_record 关联；删除考试记录时自动清理 answer_record / wrong_question
答辩重点一句话
两个 M:N 中间表解决多对多关系 + answer_record 快照字段保护历史数据 + 复合唯一约束防止重复

题库管理详情
• 题型：单选题(single)、多选题(multiple)、判断题(judge)
• 题目字段：题干(title TEXT)、A/B/C/D选项、正确答案、解析、默认分值、难度(easy/medium/hard)、知识点
• 筛选功能：按题型、难度、知识点多条件筛选 → WHERE + LIMIT 分页
• 删除方式：逻辑删除 — UPDATE question SET status = 0（保留历史考试引用）
-- 题型筛选示例
SELECT q.id, q.question_type, q.title, q.difficulty, q.knowledge_point
FROM question q WHERE q.status = 1
AND q.question_type = 'single' AND q.difficulty = 'medium'
ORDER BY q.create_time DESC LIMIT 0, 20;
试卷管理详情
• 创建试卷：设置 paper_name、duration（分钟）、start_time、end_time → INSERT INTO paper (status=0 草稿)
• 组题操作：从题库中选择题目 → INSERT INTO paper_question (paper_id, question_id, question_score, sort_order)
• 发布操作：系统自动汇总 paper_question.question_score → SUM 后写入 paper.total_score
• 发布后：学生端的"可参加考试"列表中将显示该试卷（paper.status=1 + 当前时间 BETWEEN start_time AND end_time）
-- 自动汇总总分
SELECT COALESCE(SUM(question_score), 0) FROM paper_question WHERE paper_id = #{paperId};
UPDATE paper SET status = 1, total_score = #{totalScore}, pass_score = #{passScore} WHERE id = #{paperId};

事务控制 — 提交试卷的数据一致性
@Transactional — 原子操作：全部成功或全部回滚
◎ BEGIN 开启事务
步骤 A：验证前置条件
SELECT * FROM exam_record WHERE id=?
检查：考试存在 + 属于当前学生 + 未提交 + 未超时
步骤 B：查询正确答案
SELECT pq.*, q.correct_answer
FROM paper_question pq JOIN question q
🔄 步骤 C：遍历每道题（for each question）
C1. 标准化学生答案：去空格 → 转大写 → 排序 → 去重
例："b, A" → 标准化为 "A,B"
C2. 比较正确/错误 → 正确得 question_score 分，错误得 0 分
C3. INSERT INTO answer_record（📸 保存题目快照 + 答案快照）
C4. IF 答错 → INSERT INTO wrong_question
步骤 D：汇总更新
totalScore = 各题得分累加 → UPDATE exam_record SET total_score, submit_time, status=1
◎ COMMIT 提交
全部成功
◎ ROLLBACK 回滚
任何一步失败
⚠ 回滚场景示例
假设已有 5 道题的
answer_record 保存成功
第 6 道题保存时
数据库连接中断
→ Spring 自动回滚
前 5 条 INSERT 全部撤销
exam_record 保持原值
数据一致性得到保证
事务实现原理
MySQL InnoDB 引擎：支持 ACID 事务（Atomicity 原子性、Consistency 一致性、Isolation 隔离性、Durability 持久性）
Spring @Transactional：声明式事务管理 → AOP 代理拦截方法调用 → 方法正常返回则 COMMIT → 抛出 RuntimeException 则 ROLLBACK
⚠ MyISAM 引擎不支持事务！本项目所有表使用 InnoDB 引擎，这是在第 4 页架构图中做出这一选择的原因

📋 v_student_score 视图
CREATE VIEW v_student_score AS
SELECT
er.id AS exam_id,
er.student_id,
u.real_name AS student_name,
er.paper_id, p.paper_name,
er.total_score, p.pass_score,
er.submit_time,
CASE WHEN er.total_score >= p.pass_score
THEN 1 ELSE 0 END AS is_pass
FROM exam_record er
JOIN sys_user u ON er.student_id = u.id
JOIN paper p ON er.paper_id = p.id
WHERE er.status = 1;
使用视图的优势
• 封装三表关联，简化业务查询
• is_pass 在视图层计算，避免重复 CASE WHEN
使用视图后的查询（简洁）
SELECT exam_id, student_name, paper_name, total_score, pass_score, submit_time, is_pass
FROM v_student_score WHERE student_id = ? ORDER BY submit_time DESC;
设计思路总结
索引选型原则：① 在 WHERE 条件高频出现的字段上建索引；② 在 JOIN 关联字段上建索引（如 paper_id、student_id）；
③ 唯一约束自动建立唯一索引（username、email、role_code），兼顾查询加速和数据唯一性；④ 避免在低基数字段（如 status）单独建索引
视图设计原则：封装复杂多表 JOIN + 计算字段（is_pass），上层查询简洁、可维护、逻辑统一；视图不存储数据，每次查询实时计算


关键 SQL 分析
SQL ① 查询试卷题目 — 三表关联
SELECT q.question_type, q.title, q.option_a, q.option_b,
q.option_c, q.option_d, pq.question_score, pq.sort_order
FROM exam_record er
JOIN paper_question pq ON er.paper_id = pq.paper_id
JOIN question q ON pq.question_id = q.id
WHERE er.id = ? AND er.student_id = ? AND q.status = 1
ORDER BY pq.sort_order;
📌 考点：多对多关系查询模式 — 通过中间表 paper_question 关联
SQL ② 试卷统计 — 聚合查询
SELECT p.paper_name,
COUNT(er.id) AS participant_count,
ROUND(AVG(er.total_score),2) AS avg_score,
MAX(er.total_score) AS highest_score,
MIN(er.total_score) AS lowest_score
FROM paper p
LEFT JOIN exam_record er ON er.paper_id=p.id AND er.status=1
WHERE p.id = ? GROUP BY p.id, p.paper_name;
📌 考点：COUNT/AVG/MAX/MIN + LEFT JOIN + GROUP BY
SQL ③ 可参加考试 — LEFT JOIN + 时间过滤
SELECT p.id AS paper_id, p.paper_name, p.duration,
p.total_score, p.pass_score, p.start_time, p.end_time,
er.id AS exam_id, er.status AS exam_status
FROM paper p
LEFT JOIN exam_record er ON er.paper_id=p.id AND er.student_id=?
WHERE p.status=1 AND ? BETWEEN p.start_time AND p.end_time
ORDER BY p.start_time DESC;
📌 考点：LEFT JOIN 判断是否参加过 + BETWEEN 时间范围过滤
SQL ④ 成绩查询 — 使用视图

-- 三表 JOIN + CASE WHEN，约 15 行 SQL
-- 使用视图后：
SELECT exam_id, student_name, paper_name,

📌 考点：视图封装复杂查询，上层 SQL 简洁清晰
SQL 设计能力体现
🔹 多表关联查询：三表 JOIN 通过中间表 paper_question 实现多对多关系查询 — 理解关系型数据库的核心查询模式
🔹 聚合统计查询：COUNT/AVG/MAX/MIN + GROUP BY + LEFT JOIN — 统计参加人数、平均分、最高分、最低分
🔹 时间范围查询：BETWEEN 过滤 + LEFT JOIN 判断参与状态 — 前端根据 exam_id 是否为 NULL 决定"开始"还是"继续"
🔹 视图封装：三表关联 + CASE WHEN 计算字段 → 一条 SELECT 完成成绩查询 — 体现数据库复杂查询的封装能力
